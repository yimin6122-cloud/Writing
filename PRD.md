# 灵感音域 · 产品需求文档 (PRD)

> 专为同人创作者打造的BGM氛围网站  
> 版本：v0.2-Demo → 目标 v1.0  
> 更新日期：2026-06-07（最新修订）

---

## 一、产品定位

### 一句话描述
用"IP世界观专属配乐 + 幻想氛围"激发同人创作灵感的沉浸式网站。

### 核心差异化
| 竞品（自习/专注类） | 灵感音域 |
|---|---|
| 目标：静心、专注 | 目标：**激发脑洞、催生灵感** |
| 通用白噪音（雨声、咖啡馆） | **世界观配乐**（奇幻史诗/科幻星海/古风雅韵…） |
| 无创作关联 | **配乐 ↔ 剧情/角色/场景 强关联** |
| 无社区属性 | **同好社区 + 歌单分享** |

### 目标用户
- **同人写手**：需要配乐烘托写作情绪，找符合剧情的BGM
- **同人画师**：需要氛围音乐辅助构图和色彩情绪
- **OC设定党**：需要配乐来"感受"自己角色的世界
- **剪辑师**：需要快速试听不同风格的BGM来做剪辑参考

## 三、当前技术不足（按严重程度排序）

### 🔴 P0 — 致命问题

#### 3.1 系统音乐无法满足"配乐"需求
**现状**：系统音乐使用 OfflineAudioContext 生成简单的固定和弦drone（几个正弦波/三角波叠加 + 低通滤波），听起来更像是"氛围嗡鸣"而非"音乐"。

**为什么这是问题**：
- 创作者期望的是有旋律、有情感起伏的**配乐**，不是白噪音
- 目前生成的音频没有节奏、没有旋律线、没有动态变化
- 3-4分钟的固定和弦听到30秒就腻了

**技术根因**：Web Audio API 程序化生成复杂音乐极其困难，需要：
- 多轨旋律编排（主旋律+和声+节奏+贝斯线）
- MIDI式的音符序列系统
- 或者：**接入真实的音频源**

#### 3.2 没有真实的音乐来源
**现状**：用户如果不自己上传MP3文件，就只能听程序化生成的嗡鸣声。

**为什么这是问题**：
- 这是BGM网站的**核心价值缺失**
- 大多数用户期望打开就能听
- 创作者工作流中不想花时间自己找音频文件

**可行方案**：
- 接入免费音乐API（Pixabay Audio、Jamendo、FreeSound）
- 嵌入SoundCloud Widget播放具体歌单
- 使用YouTube IFrame API播放现有的氛围音乐/OST合集
- 自建曲库（需要版权处理）

#### 3.3 白噪音功能缺失
**现状**：V1中的14种环境白噪音（雨声、风声、篝火、魔法嗡鸣等）在V2重构时完全被移除。

**为什么这是问题**：
- 白噪音是创作者"进入状态"的重要工具
- 配乐+白噪音的**叠加播放**才是完整的使用场景
- 竞品（Noisli、A Soft Murmur）的核心功能我们没有

---

### 🟡 P1 — 重要缺失

#### 3.4 创作辅助工具缺失
**现状**：V1中的灵感提示词和OC设定卡被移除，V2完全没有创作辅助功能。

**需要恢复并增强**：
- 灵感提示词生成器（场景/对白/剧情转折）
- OC人物设定卡生成器
- **新增**：配乐推荐引擎（"我写的场景是XXX，推荐BGM"）
- **新增**：创作计时器 + 字数统计 + 番茄钟

#### 3.5 首页大图是SVG占位符
**现状**：`.hero-bg` 使用内联SVG渐变代替真实图片，缺乏视觉冲击力。

**需求**：需要一张高质量的、能传达"同人创作+音乐氛围"的首页插图，或支持视频背景。

#### 3.6 世界页动态背景不够"动态"
**现状**：只有Canvas粒子动画 + 中心大emoji文字，缺乏场景感和沉浸感。

**需求**：
- 真正的动态视频背景或CSS动画场景
- 例如：奇幻世界→漂浮的魔法粒子+城堡剪影；科幻→星空流转+赛博网格
- 或者支持用户自定义背景图/GIF

#### 3.7 数据不持久化
**现状**：所有数据（上传的音乐、播放进度、用户偏好）都在内存中，刷新即丢失。

**需要**：
- `localStorage` 存储用户上传的音乐（Blob URL不行，需要用IndexedDB存文件）
- 播放历史记录
- 用户偏好（音量、循环模式、上次播放位置）
- 自定义歌单保存

---

### 🟢 P2 — 体验优化

#### 3.8 系统音乐生成性能
**现状**：
- `OfflineAudioContext` 渲染4分钟44.1kHz立体声音频在主线程执行
- 首次点击曲目需要等待1-3秒才能开始播放
- 没有生成进度提示

**优化方向**：
- 使用 `AudioWorklet` 实时合成（当前方案是离线渲染）
- 或者预生成缓存到 IndexedDB
- 或者干脆放弃程序化生成，接入真实音频源

#### 3.9 交流论坛是空壳
**现状**：论坛页面只有"即将上线"占位文字。

**即使是Demo也需要**：
- 至少有一个静态的"推荐歌单"展示区
- 嵌入现有的社区链接（LOFTER、B站、AO3等）

#### 3.10 移动端体验
**现状**：响应式布局有基础适配，但世界页在小屏幕上布局拥挤。

#### 3.11 无障碍和可访问性
- 没有 `aria-label`
- 键盘导航不完整
- 没有深色/浅色模式切换（目前只有深色）

## 四、数据架构重设计（v1.0 基础设施）

### 4.1 当前架构的问题

v0.2 的数据模型是**扁平**的——6个世界各自独立，没有层级关系：

```javascript
// 当前结构：扁平，无法支撑子世界扩展
THEMES = {
  fantasy: { name: '奇幻史诗', tracks: [...] },
  ancient: { name: '古风雅韵', tracks: [...] },
  // 每加一个子世界就要复制大量代码
}
```

用户的需求是层级化的：

```
古风雅韵（主世界）
├── ⚔️ 战场沙场  → 需要2首配乐 + 独立视觉
├── 🏯 宫廷深院  → 需要2首配乐 + 独立视觉
├── 🌸 江南园林  → 需要2首配乐 + 独立视觉
└── 🗡️ 江湖侠客  → 需要2首配乐 + 独立视觉
```

## 五、音乐源方案

### 5.1 四种方案对比

| 方案 | 音乐质量 | 稳定性 | 版权风险 | 实现难度 | Demo推荐 |
|------|---------|--------|---------|---------|----------|
| **Pixabay Music** | ⭐⭐⭐⭐ | ⭐⭐⭐⭐ | 🟢 无版权 | 简单 | ✅ **首选** |
| **增强程序化生成** | ⭐⭐ | ⭐⭐⭐⭐⭐ | 🟢 无 | 中等 | 兜底方案 |
| **YouTube IFrame** | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | 🟡 灰色地带 | 简单 | 备选 |
| **FreeSound API** | ⭐⭐⭐ | ⭐⭐⭐ | 🟢 CC授权 | 中等 | 补充 |

### 5.2 推荐策略：Pixabay为主 + 增强生成兜底

```
用户点击播放
    ↓
检查 track.sources
    ↓
首选 Pixabay URL？
    ├─ 有效 → fetch验证(3s超时) → 直接播放真实MP3
    └─ 失效/超时 → fallback
                        ↓
              增强程序化生成
              （有旋律/和弦进行/节奏层）
                        ↓
              缓存到 IndexedDB
              （下次直接播放，不再生成）
```

**Pixabay Music** 的优势：
- 有大量氛围/电影配乐/古典音乐，搜索 `epic fantasy` `ancient chinese` `sci-fi ambient` 可找到合适曲目
- 每首曲目提供直接MP3下载链接
- 完全免费、无版权、无需API Key
- 链接相对稳定

**增强程序化生成** 的改进（相比当前v0.2的固定和弦嗡鸣）：
- 🎵 **音符序列系统**：主旋律线 + 和声层 + 低音线 + 节奏层
- 🎹 **和弦进行**：如 Am→F→C→G，根据世界主题自动变化
- 🎻 **音色丰富**：不同波形叠加模拟不同乐器质感
- 📈 **动态包络**：有起承转合，音量/频率/滤波随时间演变
- ⏱️ **时长可配**：生成2-5分钟完整曲目

### 5.3 实操方式

每个世界/子世界由我手动在 Pixabay 上挑选2首合适的曲目，提取直接MP3链接写入配置。同时保留增强程序化生成作为离线兜底。用户上传功能继续保留。

## 七、推荐开发路径（6步）

**为什么是这个顺序？** 每一步都是下一步的基石，避免返工：

| 步骤 | 内容 | 产出 | 为什么先做这个 |
|------|------|------|---------------|
| **1. 数据架构重构** | 改为JSON配置驱动的层级结构 | 支持无限扩展的配置体系 | 决定了后面所有代码怎么写；不先做的话加子世界要改5处 |
| **2. 音乐管线改造** | Pixabay真实MP3接入 + 增强程序化生成引擎 + 多源fallback + IndexedDB缓存 | 每个世界至少2首可播放的真音乐 | 音乐是核心价值，必须在UI之前跑通 |
| **3. 子世界导航UI** | 世界页内横向标签切换、联动歌单/白噪音/提示词 | 用户可以在父子世界间流动 | 有内容但找不到入口就是白做 |
| **4. 子世界内容填充** | 为古风4个子世界 + 奇幻3个子世界各配2首Pixabay音乐 + 专属提示词 | 7×2=14首真实曲目 | 基于1-3的框架填入数据 |
| **5. 旧世界迁移** | 把其余主世界也纳入新架构，每世界2首音乐 | 全站统一、无遗留代码 | 新旧两套代码并存是维护灾难 |
| **6. 恢复&打磨** | 恢复白噪音混音器 + 灵感提示词 + OC设定卡；视觉打磨 | 完整创作体验 | 基于完整功能做优化，提前做可能白费 |

### 各步骤详解

#### 第1步：数据架构重构
- 创建 `WORLD_TREE` 配置对象（层级化）
- 改造 `buildPlaylist()` 等函数适配新结构
- 保持现有UI不变，先让数据层跑通
- **验收标准**：console.log(WORLD_TREE) 输出完整层级，现有6个世界仍然能正常导航

#### 第2步：音乐管线改造
- 实现 `resolveTrackSource(track)` 函数：先试Pixabay URL，3秒超时则fallback
- 重写增强程序化生成：音符序列+和弦进行+多轨编排
- 实现 `IndexedDB` 缓存层（生成过的曲目永久缓存）
- **验收标准**：点击每个世界的曲目，3秒内开始播放真实音乐

#### 第3步：子世界导航UI
- 世界页顶部增加横向滚动的子世界标签栏
- 标签切换时联动更新背景、歌单、白噪音
- 过渡动画
- **验收标准**：点击子世界标签，视觉+音乐在0.5秒内切换

#### 第4步：子世界内容填充
- 在Pixabay为古风4个子世界 + 奇幻3个子世界挑选曲目
- 配置专属提示词、白噪音预设
- **验收标准**：7个子世界各有2首可播放音乐+完整配置

#### 第5步：旧世界迁移
- 其余4个主世界（科幻/都市/末日/校园）各配置2首Pixabay音乐
- 每个主世界规划子世界框架（先有配置结构，内容后续填充）
- 清理旧代码中的 `THEMES` 常量
- **验收标准**：全站使用统一的 `WORLD_TREE`，无遗留代码

#### 第6步：恢复功能 & 打磨
- 恢复白噪音混音器（适配新架构）
- 恢复灵感提示词+OC设定卡
- 移动端适配优化
- 首页视觉增强

## 九、技术方案总结

| 问题 | 推荐方案 | 备选方案 |
|------|---------|---------|
| 真实音乐源 | Pixabay直链MP3（手动精选） | SoundCloud Widget |
| 音乐兜底 | 增强程序化生成（音符序列+和弦+多轨） | 当前固定和弦drone |
| 数据持久化 | IndexedDB存生成音频缓存 + localStorage存偏好 | — |
| 数据架构 | 层级JSON配置驱动（WORLD_TREE） | 当前扁平硬编码 |
| 白噪音生成 | Web Audio API 实时噪声发生器 | 预录WAV文件 |
| 动态背景 | CSS动画 + Canvas合成 | 视频文件 |
| 音乐可视化 | AnalyserNode + Canvas（已实现） | Three.js 3D可视化 |
| 前端框架 | **保持纯HTML/CSS/JS** | React/Vue（如扩展） |

## 十、非功能需求

- **性能**：页面加载 < 3s，音频切换延迟 < 500ms，首次生成程序化音乐 < 3s
- **兼容性**：Chrome/Safari/Firefox/Edge 最新两个版本
- **可访问性**：WCAG 2.1 AA 标准（后续补充 aria-label 等）
- **SEO**：语义化HTML + meta标签（为后续上线做准备）
- **隐私**：不上传用户本地文件到任何服务器（纯前端处理）
- **离线能力**：程序化生成音乐可离线工作；Pixabay曲目需联网但可缓存
